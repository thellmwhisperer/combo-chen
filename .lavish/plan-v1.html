The pipeline

DIRECTOR  (orchestrates & watches; NEVER touches code)        [tmux: combo-chen-N]
   │
   ├─ PHASE 1 · ROWER     gnhf loop in a treehouse worktree   → thread_id captured
   ├─ PHASE 2 · HODOR     no-mistakes: review→test→docs→lint→push→PR
   ├─ PHASE 3 · REVIEW    reviewer(s) on /loop  ⇄  RESUMED rower addresses threads
   │                      hodor ci-step in parallel: CI red / conflicts → auto-fix
   └─ READY → MERGE       human (hard default); automerge earned per risk tier

Hard rules: reviewer ≠ rower (validated at launch) · LGTM pinned to SHA, expires on push · push semaphore (rower never pushes while a hodor CI-fix is in flight) · rate limits are events, not failures.

The two babysitters (investigated boundary)

hodor no-mistakes ci-step · "hold the door"

Machine signals: watches PR until merge/close, auto-fixes CI failures & merge conflicts (force-push). Verified: never reads or answers review threads.

thread-sitter the resumed rower — combo-chen's contribution

Conversation: the SAME thread that implemented, resumed (codex resume / hermes --resume / ACP). Mechanical comments → handled autonomously. Intent-touching → needs_human, thread pauses.

From Kun's interview → spec deltas

  • Preflight: the issue IS the spec; plan quality buys autonomous runtime → combo-chen preflight grades the issue before launch.
  • Fresh-context reviewing is empirical: resumed context for WRITING, fresh context for JUDGING.
  • Risk-tier automerge: the counterfactual log keys on the gate's risk assessment; low-risk earns trust first.
  • Attention surface: tmux titles + combo-chen status always answer "which combos need a human right now".
